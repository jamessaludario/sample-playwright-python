Additional info

Debugging tools built into Playwright

  • Codegen (test recorder). Opens a browser; as you click around it writes Python code for every action. Great for learning locator syntax: 
    playwright codegen https://automationexercise.com
  • Playwright Inspector (step-through debugger). Runs your test one action at a time with a live locator explorer: 
    # Windows (PowerShell)
$env:PWDEBUG = "1"; pytest tests/test_login.py

# macOS / Linux
PWDEBUG=1 pytest tests/test_login.py
  • Trace Viewer (flight recorder). Run with --tracing retain-on-failure and each failure saves a trace.zip. Open it to scrub through every step with before/after screenshots, console logs, and network calls: 
    playwright show-trace path/to/trace.zip
    This is the single most powerful way to diagnose a failure that only happens on the CI server.

Dealing with flaky tests

A flaky test passes sometimes and fails sometimes without any code change — usually caused by timing (the site was slow) or shared state (two tests fighting over the same account). Defenses, in order of preference:

  1. Replace any fixed waits with expect() assertions (Section 7).
  2. Make every test create its own data — e.g. register a fresh user with a unique, timestamped email instead of sharing one account.
  3. As a safety net, auto-retry failures with pytest-rerunfailures: --reruns 2 --reruns-delay 2. Retried tests are flagged in reports, so you can still spot and fix the flakiness rather than hide it.

Generating unique test data

utils/data.py 
import uuid


def unique_email() -> str:
    """A fresh email every call, e.g. qa-3f2a1c@example.com,
    so tests never collide over the same account."""
    return f"qa-{uuid.uuid4().hex[:6]}@example.com"

Good habits checklist

  • Keep all locators in page objects — never in tests.
  • Wait with expect(), never time.sleep().
  • One behavior per test; name tests after that behavior.
  • Tests create their own data and clean up after themselves.
  • Register markers and use --strict-markers.
  • Add .venv/, reports/, and __pycache__/ to .gitignore.
  • Start small: automate 5 solid smoke tests before attempting 50.

Running tests automatically (CI) — a peek ahead

Once your suite is stable, teams wire it into Continuous Integration (CI) — a service like GitHub Actions or Jenkins that runs the tests automatically on every code change and posts the report. Playwright runs headless on CI servers out of the box, so the suite you built locally is already CI-ready. That's a topic for its own guide, but know that it's the natural next step.
