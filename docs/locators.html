How to access elements (locators)

What is a locator?

A web page is built from HTML elements — buttons, text boxes, links, images. Before your script can click a button, it must first find that button among the hundreds of elements on the page. A locator is the "address" you give Playwright so it can find the right element.

Think of it like giving directions: "the button labeled Login" or "the text box with placeholder Email Address". In code: 

# Create a locator (nothing happens on the page yet - it's just the address)
login_button = page.get_by_role("button", name="Login")

# Use it - NOW Playwright finds the element and clicks it
login_button.click()
Locators are "lazy" and self-updating A locator doesn't search the page when you create it — only when you act on it (.click(), .fill(), or inside expect()). Each action re-finds the element, so locators keep working even on pages that redraw themselves.

The recommended locators (use these first)

Playwright's get_by_* methods find elements the way a human would — by their visible label, role, or placeholder. They survive code refactors much better than locators tied to HTML internals.

LocatorFinds elements by...Example
get_by_role() Their role (button, link, heading, checkbox...) and visible name. The best default choice. page.get_by_role("button", name="Sign up")
get_by_text() The text they display. page.get_by_text("Welcome back")
get_by_label() The form label next to an input. page.get_by_label("Password")
get_by_placeholder() The gray hint text inside an empty input. page.get_by_placeholder("Email Address")
get_by_title() Their title attribute (tooltip text). page.get_by_title("Close dialog")
get_by_alt_text() An image's alternative text. page.get_by_alt_text("Company logo")
get_by_test_id() A data-testid attribute developers add specifically for tests. Very stable — ask your developers to add them! page.get_by_test_id("submit-order")

CSS and XPath locators (when the above aren't enough)

page.locator() accepts CSS selectors (the same syntax used to style pages) and XPath. You'll need these for elements without good labels: 

# CSS: by id (the # symbol) - ids should be unique on a page
page.locator("#susbcribe_email")

# CSS: by class (the . symbol)
page.locator(".shopping-cart")

# CSS: by tag + attribute
page.locator("input[name='search_product']")

# CSS: element inside another element (a space means "descendant")
page.locator("#cart_items .product-name")

# XPath (starts with // ) - powerful but harder to read; prefer CSS
page.locator("//button[text()='Checkout']")

How to discover an element's address

  1. Open the page in your browser.
  2. Right-click the element and choose Inspect. DevTools opens with its HTML highlighted, e.g. <button class="btn btn-primary" data-qa="login-button">Login</button>
  3. Pick the most stable, human-readable address: here, role button with name Login, or the data-qa attribute via page.locator("[data-qa='login-button']").

Or let Playwright suggest one for you — its recorder shows a recommended locator for anything you hover:

Narrowing down: chaining and filtering

When several elements match, get more specific:

# Search INSIDE another element (chaining)
page.locator(".navbar").get_by_role("link", name="Products")

# Keep only matches that contain certain text (filtering)
page.get_by_role("listitem").filter(has_text="Blue Top")

# Pick by position - use as a last resort, order can change
page.get_by_role("link", name="View Product").first
page.get_by_role("row").nth(2)          # third match (counting starts at 0)

Common actions on a located element

ActionWhat it does
.click()Click it.
.fill("text")Clear a text box and type the text.
.press("Enter")Press a keyboard key while focused on it.
.check() / .uncheck()Tick / untick a checkbox.
.select_option("USA")Choose from a dropdown (<select>).
.hover()Move the mouse over it (to open hover menus).
.inner_text()Read the text it displays.
.scroll_into_view_if_needed()Scroll until it is on screen.
Priority order when choosing a locator 1) get_by_role  2) other get_by_* methods  3) get_by_test_id  4) simple CSS  5) XPath (last resort). The higher on this list, the less likely your test breaks when the page's internals change.
